## CodeIgniter 4 Production Deployment Rules & Workflow

### 1. AI Agent Workflow

1.  **Analyze Request**: Deconstruct user requests into logical, sequential file modifications that align with the CodeIgniter 4 file structure.
2.  **Propose Changes**: Before implementation, clearly state the intended modifications and the files to be affected. For example: "I will add a `description` column to the `products` table via a new migration, update the `ProductModel` and its Entity in `app/Models/Product.php`, modify the `Products` controller in `app/Controllers/Products.php`, and add the field to the `product_form` view in `app/Views/products/form.php`."
3.  **Adhere to Architectural Principles**: All modifications must respect the CodeIgniter 4 file structure and its separation of concerns.
    *   **Configuration (`app/Config/`)**: All core application settings are managed here. Modifications should target specific files like `App.php`, `Database.php`, `Routes.php`, etc.
    *   **Controllers (`app/Controllers/`)**: Handle HTTP requests. Keep them lean and focused on request/response flow. **All business logic must be delegated to Models or Libraries/Services** to ensure scalability and maintainability.
    *   **Models (`app/Models/`)**: Manage all database interactions. Use CodeIgniter's Model class, Query Builder, or **Entities** for a robust data layer. Never place direct database queries in Controllers. (See: https://codeigniter.com/user_guide/models/model.html)
    *   **Views (`app/Views/`)**: Handle presentation. They should contain minimal PHP and primarily display data passed from controllers. All dynamic data must be escaped with `esc()` to prevent XSS attacks. For efficiency and maintainability, **use View Layouts for page structure and View Cells for reusable components**. (See: https://codeigniter.com/user_guide/outgoing/view_layouts.html)
    *   **Database Migrations & Seeds (`app/Database/`)**: Schema changes must be handled through Migration files. Initial data for development or testing should be managed with Seeder files.
    *   **Filters (`app/Filters/`)**: Use filters judiciously for cross-cutting concerns. They should be restricted to essential functionality like security (`csrf`, `honeypot`), authentication, or rate limiting (`throttler`). Custom business logic does not belong in filters. (See: https://codeigniter.com/user_guide/incoming/filters.html)
    *   **Helpers (`app/Helpers/`)**: Contain stateless, procedural helper functions.
    *   **Language (`app/Language/`)**: Store language-specific strings for internationalization.
    *   **Libraries (`app/Libraries/`)**: House custom classes that provide reusable application logic. For better scalability and testability, these should be managed as **Services** in `app/Config/Services.php`. (See: https://codeigniter.com/user_guide/concepts/services.html)
4.  **Prioritize Framework Functions**: Before writing custom helper functions or libraries, exhaust all built-in CodeIgniter functionalities. This approach reduces code footprint, simplifies maintenance, and leverages the framework's stability.
5.  **File Creation & Modification**:
    *   **File Creation**: All new boilerplate files—including Controllers, Models, Migrations, Seeds, and Entities—**must** be generated using the appropriate `php spark` command (e.g., `php spark make:controller Products --restful`, `php spark make:model Product --table products --return entity`, `php spark make:migration CreateProductsTable`). Manual creation of these files is forbidden. Use command flags to generate more complete boilerplate.
    *   **Preserve Boilerplate Structure**: The file structure, class name, namespace, and boilerplate methods generated by `php spark` commands **must** be preserved. Only add the code necessary to implement the specific task. Do not alter or remove the framework-provided template code.
    *   **File Modification**: When providing code for a new or modified file, always specify the full path from the project root (e.g., `FILE: app/Controllers/Home.php`). Provide the complete, updated file content.
6.  **Final Confirmation**: Before concluding, confirm that all changes have been applied in accordance with the established rules and file structure.

### 2. Environment Configuration

1.  The `CI_ENVIRONMENT` variable in the `.env` file **must** be set to `production`.
2.  The production `.env` file **must not** be committed to version control. A template file, `env` (which should be copied to `.env`), may be used as a placeholder.
3.  All environment-specific variables, including database credentials, API keys, and other secrets, **must** be defined in the `.env` file and accessed via `env()`. They should not be hardcoded in configuration files within `app/Config/`.
4.  The application's base URL (`app.baseURL`) **must** be correctly set in the `.env` file, with a trailing slash (e.g., `app.baseURL = 'https://example.com/'`).
5.  **Prioritize Core Configuration**: For settings that define application structure or behavior (e.g., filter aliases, module discovery, validation rule sets), modifications **must** be made directly in the relevant `app/Config/` file (e.g., `Filters.php`, `Modules.php`, `Validation.php`). The `.env` file should be reserved for variables that change per environment, not for overwriting core architectural settings.

### 3. Web Server & Filesystem Configuration

1.  The web server's document root **must** be pointed to the project's `/public` directory.
2.  For security, the `app`, `system`, and `writable` directories **must** be located outside of the web server's document root.
3.  The `writable` directory and its subdirectories **must** have write permissions for the web server user.
4.  Development-only files and directories **must** be removed from the production server. This includes the `tests/` directory, `phpunit.xml.dist`, `spark`, and `preload.php`.

### 4. Code & Dependency Management

1.  Before deployment, execute `composer install --no-dev --optimize-autoloader` from the project root to install only production dependencies and optimize the class autoloader.
2.  All unused modules in **app/Config/Modules.php** should be disabled to reduce auto-discovery overhead.
3.  **Adhere to Coding Standards**: All PHP code **must** follow the PSR-12 coding standard. Use modern PHP features, including strict types (`declare(strict_types=1);`), typed properties, and return type declarations to improve code quality and maintainability.
4.  **Leverage Service Injection**: For scalability, custom classes providing services **must** be defined in `app/Config/Services.php`. This allows the application to manage dependencies efficiently and makes components easier to test and swap.
5.  **API Development**: For API endpoints, use CodeIgniter’s **API Response Trait** to standardize responses. Utilize methods like `respond()`, `fail()`, and appropriate HTTP status codes. (See: https://codeigniter.com/user_guide/outgoing/api_responses.html)

### 5. Error Handling & Logging

1.  Detailed error reporting to the browser is disabled by default in the production boot file (`app/Config/Boot/production.php`); this setting **must not** be altered.
2.  Exception logging **must** be enabled by setting `$log` to `true` in **app/Config/Exceptions.php**.
3.  The logging threshold in **app/Config/Logger.php** **must** be configured to an appropriate level for production (e.g., a threshold of `4` to log only critical errors). Logs are written to `writable/logs/`.
4.  To ensure a user-friendly experience, custom error pages for common HTTP status codes (e.g., 404, 503) **must** be created in **app/Views/errors/html/**.

### 6. Performance Optimization

1.  For applications with explicitly defined routes in `app/Config/Routes.php`, disable "Auto Routing" by setting `$autoRoute` to `false` in **app/Config/Routing.php**.
2.  For data that is accessed frequently but changes infrequently, enable and configure a production-ready caching driver like Redis or Memcached in **app/Config/Cache.php**.
3.  As part of the deployment script, use the `php spark optimize` command to enable config and file locator caching. This requires the cache to be manually cleared (`php spark cache:clear`) after any configuration or file path changes.
4.  **Optimize Database Queries**: To reduce memory footprint, always select only the necessary columns using `$builder->select('col1, col2')`. Avoid using `findAll()` where a more specific method like `find()` or a custom query with a `limit` can be used.

### 7. Security Mandates (Non-Negotiable)

1.  **CSRF Protection**: Cross-Site Request Forgery (CSRF) protection **must** be enabled globally in **app/Config/Filters.php**. All web forms must include the CSRF token.
2.  **Content Security Policy (CSP)**: To mitigate Cross-Site Scripting (XSS) attacks, CSP **must** be enabled (`$CSPEnabled = true` in **app/Config/App.php**) and configured in **app/Config/ContentSecurityPolicy.php**.
3.  **Database Interactions**: All database queries **must** utilize the Query Builder or prepared statements to prevent SQL injection. Raw queries are forbidden.
4.  **Input Validation**: All user-submitted data (`$_GET`, `$_POST`, etc.) **must** be validated with the CodeIgniter Validation library before being processed.
5.  **Output Escaping**: All data rendered in HTML views **must** be escaped using the `esc()` function to prevent XSS attacks.
6.  **Session Security**: When using HTTPS, the `cookie.secure` and `cookie.httponly` settings in your `.env` file **must** be set to `true`. For enhanced security, session data should be stored in a database (`DatabaseHandler` in `app/Config/Session.php`).
7.  **Rate Limiting**: To prevent brute-force attacks on login forms or API endpoints, the **Throttler** filter **must** be enabled and configured on the relevant routes. (See: https://codeigniter.com/user_guide/libraries/throttler.html)

### 8. Development & Testing

1.  **Unit & Feature Testing**: All new business logic in models, libraries, or services **should** be accompanied by corresponding tests in the `tests/` directory. Use CodeIgniter's test case classes and `php spark` commands to generate test files. A robust test suite is essential for long-term maintainability. (See: https://codeigniter.com/user_guide/testing/index.html)
